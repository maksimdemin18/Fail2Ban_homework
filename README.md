### Решение:

Разведка сервисов Nmap и анализ логов Suricata / Fail2Ban

nmap -sA — это ACK scan: он помогает понять, какие порты фильтруются, и не определяет их как open; *-sT — TCP connect scan* с полноценным установлением соединения; *-sS — SYN scan; -sV — service/version detection*, то есть дополнительный опрос найденных портов для определения протокола, имени сервиса и версии. 

Suricata по умолчанию пишет журналы в */var/log/suricata*; основные файлы для отчёта — fast.log с однострочными alert-событиями и eve.json с JSON-событиями (alert, flow, anomaly и др.). 

Fail2Ban анализирует журналы аутентификации и при множественных ошибках входа применяет бан; jails по умолчанию отключены (enabled = false), а для SSH в конфигурации есть jail [sshd] с параметрами port, logpath и backend. 

#### 1.1. Выполнение разведки

На системе злоумышленника выполним:
```
sudo nmap -sA 192.168.18.144 | tee nmap_sA.txt
sudo nmap -sT 192.168.18.144 | tee nmap_sT.txt
sudo nmap -sS 192.168.18.144 | tee nmap_sS.txt
sudo nmap -sV 192.168.18.144 | tee nmap_sV.txt
```

#### 1.3. Что ожидать в результате

Для этапа nmap Fail2Ban чаще всего ничего не фиксирует, и это нормальный результат: он реагирует не на сам факт порт-сканирования, а на многократные ошибки аутентификации в отслеживаемых логах. 

Поэтому отсутствие записей Fail2Ban после nmap — корректное наблюдение. 

Suricata может:

показать alert-события по сканированию, если в загруженном ruleset есть соответствующие сигнатуры;

не показать alert-событий вообще, если подходящих правил нет или трафик не попал под детектирование. 

Это тоже допустимо, потому что fast.log хранит именно alert-события, а eve.json — события тех типов, которые включены в конфигурации. 

Это уже вывод из того, как устроены выходы Suricata. 

1.4. Вывод

В ходе разведки защищаемой системы были выполнены сканы *nmap -sA, nmap -sT, nmap -sS и nmap -sV*. 

ACK-скан -sA использовался для оценки фильтрации портов и состояния межсетевого экрана и не предназначен для определения портов как открытых. 

TCP connect scan -sT выполнял полное TCP-соединение, SYN scan -sS осуществлял полуоткрытое сканирование, а -sV дополнительно использовался для определения сетевых служб и их версий.

В логах Suricata были зафиксированы события, связанные с анализом сетевого трафика. 

В файле eve.json присутствовали события типов *flow, dhcp, mdns, ssh и stats*, что подтверждает корректную обработку сетевых пакетов системой Suricata. 

Кроме того, в *fast.log и eve.json* были получены alert-события с сигнатурой LOCAL TCP scan test, сформированные при выполнении TCP-сканирования защищаемого узла. 

В частности, в журналах были зафиксированы обращения с *IP-адреса 192.168.18.142* к защищаемому хосту *192.168.18.144* на различные TCP-порты, включая порт 22, что свидетельствует о регистрации сканирующей активности системой обнаружения вторжений.

В логах Fail2Ban после этапа разведки событий, связанных с блокировкой адреса атакующей системы, зафиксировано не было. 

Это является ожидаемым результатом, поскольку Fail2Ban предназначен для анализа событий аутентификации и реагирует на повторяющиеся ошибки входа, а не на обычное сканирование портов.

Таким образом, в ходе выполнения задания было установлено, что Suricata успешно фиксирует сетевую активность и позволяет выявлять признаки сканирования при наличии соответствующих правил обнаружения, тогда как Fail2Ban на этапе разведки, как правило, не формирует событий. 

Следовательно, Suricata выполняет функцию сетевого мониторинга и обнаружения подозрительного трафика, а Fail2Ban более эффективен на этапе атак, связанных с подбором паролей и неудачными попытками аутентификации.

### Решение:

Подбор пароля к SSH через Hydra и анализ логов


#### 2.1. Подготовка Fail2Ban для SSH

Откроем /etc/fail2ban/jail.conf, найдем секцию SSH и включим её. 

В конфигурации Fail2Ban jails по умолчанию выключены

```
[sshd]
enabled = true
port = ssh
logpath = %(sshd_log)s
backend = %(sshd_backend)s
```

После этого перезапустим сервис:
```
sudo systemctl restart fail2ban
sudo systemctl status fail2ban --no-pager
sudo fail2ban-client status
sudo fail2ban-client status sshd
```

#### 2.2. Подготовка файлов users.txt и pass.txt

Пример:
```
cat > users.txt <<'EOF'
root
admin
test
student
realuser
EOF
```
```
cat > pass.txt <<'EOF'
123456
password
qwerty
admin
realpassword
EOF
```


#### 2.3. Выполнение атаки

Запустим команду:
```
hydra -L users.txt -P pass.txt 192.168.18.14 ssh
```

2.6. Вывод:

Для имитации атаки подбора пароля к службе SSH были созданы файлы *users.txt и pass.txt*, после чего выполнена команда *hydra -L users.txt -P pass.txt <IP> ssh*. 

На защищаемой системе предварительно была включена SSH-защита Fail2Ban через секцию [sshd] с параметром *enabled = true*. 

В ходе эксперимента было подтверждено, что Hydra осуществляет множественные попытки подключения к SSH, а *Fail2Ban* отслеживает ошибки аутентификации и автоматически блокирует IP-адрес источника при превышении порога неудачных попыток.

В логах *Suricata* также были зафиксированы события, связанные с атакующей активностью в отношении SSH-службы защищаемого узла.

Таким образом, в ходе выполнения задания было подтверждено, что Fail2Ban эффективно защищает SSH-службу от атак подбора пароля, реагируя на повторяющиеся ошибки аутентификации и блокируя IP-адрес атакующей системы. 

Suricata, в свою очередь, зафиксировала сетевую активность, связанную с обращениями к SSH-порту и аномальным поведением TCP-потока.
