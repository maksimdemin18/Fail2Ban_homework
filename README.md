### Решение:

Разведка сервисов Nmap и анализ логов Suricata / Fail2Ban

nmap -sA — это ACK scan: он помогает понять, какие порты фильтруются, и не определяет их как open; -sT — TCP connect scan с полноценным установлением соединения; -sS — SYN scan; -sV — service/version detection, то есть дополнительный опрос найденных портов для определения протокола, имени сервиса и версии. 

Suricata по умолчанию пишет журналы в /var/log/suricata; основные файлы для отчёта — fast.log с однострочными alert-событиями и eve.json с JSON-событиями (alert, flow, anomaly и др.). 

Fail2Ban анализирует журналы аутентификации и при множественных ошибках входа применяет бан; jails по умолчанию отключены (enabled = false), а для SSH в конфигурации есть jail [sshd] с параметрами port, logpath и backend. 

1.1. Выполнение разведки

На системе злоумышленника выполните:

sudo nmap -sA <IP_защищаемой_системы>
sudo nmap -sT <IP_защищаемой_системы>
sudo nmap -sS <IP_защищаемой_системы>
sudo nmap -sV <IP_защищаемой_системы>

Для аккуратного отчёта лучше сохранить выводы в файлы:

sudo nmap -sA <IP> | tee nmap_sA.txt
sudo nmap -sT <IP> | tee nmap_sT.txt
sudo nmap -sS <IP> | tee nmap_sS.txt
sudo nmap -sV <IP> | tee nmap_sV.txt

1.2. Снятие логов на защищаемой системе

После выполнения сканов снимите события Suricata:

sudo tail -n 50 /var/log/suricata/fast.log
sudo jq 'select(.event_type=="alert")' /var/log/suricata/eve.json | tail -n 20

Если jq не установлен:

sudo tail -n 50 /var/log/suricata/eve.json

Для Fail2Ban:

sudo tail -n 50 /var/log/fail2ban.log
sudo fail2ban-client status
sudo fail2ban-client status sshd

1.3. Что честно ожидать в результате
Для этапа nmap Fail2Ban чаще всего ничего не фиксирует, и это нормальный результат: он реагирует не на сам факт порт-сканирования, а на многократные ошибки аутентификации в отслеживаемых логах. Поэтому отсутствие записей Fail2Ban после nmap — корректное наблюдение. 

Suricata может:

показать alert-события по сканированию, если в загруженном ruleset есть соответствующие сигнатуры;

не показать alert-событий вообще, если подходящих правил нет или трафик не попал под детектирование. Это тоже допустимо, потому что fast.log хранит именно alert-события, а eve.json — события тех типов, которые включены в конфигурации. Это уже вывод из того, как устроены выходы Suricata.

### Решение:

Задание 2. Подбор пароля к SSH через Hydra и анализ логов

Hydra поддерживает SSH и в официальном README прямо указано, что инструмент предназначен только для легального использования в исследовательских и тестовых целях. 

2.1. Подготовка Fail2Ban для SSH

Строго по заданию откройте /etc/fail2ban/jail.conf, найдите секцию SSH и включите её. В конфигурации Fail2Ban jails по умолчанию выключены; пример включения SSH jail через enabled = true показан в типовом jail.conf, а секция [sshd] содержит параметры для SSH-сервиса. 

Измените:

[sshd]
enabled = true
port = ssh
logpath = %(sshd_log)s
backend = %(sshd_backend)s

После этого перезапустите сервис:

sudo systemctl restart fail2ban
sudo systemctl status fail2ban --no-pager
sudo fail2ban-client status
sudo fail2ban-client status sshd

2.2. Подготовка файлов users.txt и pass.txt

Пример:

cat > users.txt <<'EOF'
root
admin
test
student
realuser
EOF

cat > pass.txt <<'EOF'
123456
password
qwerty
admin
realpassword
EOF

Если хотите показать срабатывание максимально наглядно, можно добавить одного реального пользователя и один правильный пароль, но только в учебной среде.

2.3. Выполнение атаки
Запустите команду ровно в форме задания:

На защищаемой системе сразу снимите события:

sudo tail -n 50 /var/log/fail2ban.log
sudo fail2ban-client status sshd
sudo tail -n 50 /var/log/suricata/fast.log
sudo jq 'select(.event_type=="alert")' /var/log/suricata/eve.json | tail -n 20

2.5. Что честно ожидать в результате

Для SSH-подбора Fail2Ban обычно должен зафиксировать события вида Found и затем Ban, если число неудачных попыток превысило порог jail. Это соответствует логике Fail2Ban: он читает журналы аутентификации и применяет бан к источнику повторных ошибок входа. 

Suricata на этапе Hydra может:

зафиксировать SSH-события или brute-force/scanning alert-события, если нужные правила загружены;

не показать alert-записей, если правила на SSH brute force в текущем ruleset отсутствуют. Пустой fast.log в этом месте не означает, что атаки не было; это означает только, что не было совпавших alert-сигнатур.
